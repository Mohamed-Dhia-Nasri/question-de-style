## Current cost model (verified prices)

Assumptions: 30-day month, default cadence (4 full cycles/day = 120/mo; 6 story-only + 4 full-cycle story dispatches = 10 story runs/day per the code audit), content limit 30, one handle per run (no batching), no date filters. All per-event prices below were VERIFIED against live Apify pages/API on 2026-07-07. What is ESTIMATED: posting behavior, whether the story actor bills its $0.099 start on access-error runs, and the TikTok 'filter-applied' emission granularity (not used at defaults).

## Recommendations (ranked)

### 1. Incremental date-window fetch with a bounded metric-refresh window (biggest single lever)

- **Savings:** 50-85% of content spend, i.e. roughly $250-400/mo off the $530 roster baseline, depending on posting rate. Arithmetic: content is re-billed 3,600 items/mo per IG creator ($19.20) and 3,600/mo per TikTok creator ($13.44 incl. starts) regardless of activity. With a 14-day window, a creator with ~10 items in-window bills ~1,200 items/mo instead of 3,600 (-67%); a quiet creator posting weekly bills ~240 (-93%). TikTok's date filter adds the verified 'filter-applied' event (+$0.0013/result Free tier, so $0.0050/item), still an order-of-magnitude net cut (research-verified example: $0.1167 -> ~$0.005-0.011 per creator per cycle).
- **Effort:** small code change + needs ADR/doc note (input params change; actors and SRC ids unchanged, so within ADR-0001/0002 governance — same precedent as the story-actor env deviation)
- **Changes:** app/Platform/Ingestion/Providers/Instagram/InstagramPostAdapter.php:49-53 and InstagramReelAdapter.php:45-46: add verified param 'onlyPostsNewerThan' (accepts ISO or relative, e.g. '14 days') plus 'skipPinnedPosts': true (verified mitigation for the documented pinned-post leak). app/Platform/Ingestion/Providers/TikTok/TikTokContentAdapter.php:47-50: add verified param 'oldestPostDateUnified' => '14 days' (keep default profileSorting 'latest' — date filters verified to only work with latest/oldest). New config key qds.ingestion.refresh_window (env QDS_INGESTION_REFRESH_WINDOW) alongside content_results_limit at config/qds.php:60; keep resultsLimit 30 as a safety cap.
- **Risks / mitigation:** KPI risk: posts older than the window stop getting metric refreshes, so a late viral surge on an old post would freeze in the EMV/engagement time series. Mitigate: (a) window >= the engagement growth horizon (14 days covers the vast majority of IG/TikTok engagement accrual); (b) one full-depth sweep per week (single cycle without the date filter) to catch late bloomers; (c) TikTok 'filter-applied' billing granularity is undocumented — the savings math already assumes the worst case (per result).

### 2. Stop the doomed story polling: remove story dispatch from full cycles, add a story kill-switch and provider circuit breaker

- **Savings:** Today: eliminates ~300 guaranteed-failing actor invocations + 300 ProviderCall failure rows + alert churn per IG creator per month (dollar spend today is ~$0 only because the allowlist rejects the free account — an accident, not a control; on a paid account these could bill $0.099/run = up to $30.60/mo/creator of pure waste, unverified). Permanently: fixes the audit-confirmed 10-runs/day bug (4 full-cycle + 6 story-cycle, 30 minutes apart at 00:00/00:30 and 12:00/12:30) down to the documented 6 — a 40% story-run cut whenever stories go live.
- **Effort:** small code change
- **Changes:** app/Platform/Ingestion/Jobs/PollMonitoredAccountJob.php:71-73 — stop dispatching IngestStoriesJob in the full-cycle branch (stories belong to the '30 */4 * * *' cycle only); app/Platform/Ingestion/Support/PollPlan.php:25 — drop the '+1 if hasStories' from full-cycle plans; add config qds.ingestion.stories_enabled (env) so the paid actor can be switched off without editing crons; extend app/Platform/Ingestion/Observability/ProviderCallRecorder.php:190-217 so a FAILING ProviderHealthState gates the next dispatch (currently alert-only) instead of re-invoking a permanently dead provider forever.
- **Risks / mitigation:** Essentially none to KPI today (every story run already fails). The circuit breaker needs a recovery probe (e.g. one canary call per N cycles) so a provider that comes back is re-enabled automatically rather than staying dark.

### 3. Batch story polling into one run per cycle and switch that call to the async run endpoint

- **Savings:** 92% of story spend once stories are enabled. Arithmetic at 20 IG creators, 6 polls/day, verified prices ($0.099/run start + $0.003/username): unbatched = 20 x 6 x 30 x $0.102 = $367.20/mo; batched (one run, 20 usernames — actor input verified to accept up to 100/run) = 6 x 30 x ($0.099 + 20 x $0.003) = 180 x $0.159 = $28.62/mo. The $0.099 start fee is the dominant story cost and only batching amortizes it.
- **Effort:** small-to-medium code change + ADR/doc note (endpoint change; the story actor is already a documented deviation, and access requires either the $29 Starter plan or emailing datavoyant@gmail.com for the free allowlist)
- **Changes:** Story adapter: send the whole IG roster in one 'usernames' array (verified maxItems 100) instead of {usernames:[handle]} per account. app/Platform/Ingestion/Http/ApifyClient.php: add an async path (POST /v2/acts/{id}/runs, then poll waitForFinish<=60s / fetch dataset items) for batched runs — the sync endpoint hard-408s at 300s (verified) and the client's APIFY_TIMEOUT_SECONDS is only 240 (config/services.php:49), and an aborted-but-completed sync run is still billed. Restructure IngestStoriesJob from per-account to per-cycle.
- **Risks / mitigation:** One failed batched run loses that cycle's stories for the whole roster (vs one account) — acceptable given 6 cycles/day and 24h story lifetime; keep tries with backoff. Per-account ProviderCall attribution needs a fan-out step when persisting the batched dataset. Alternative actor note: for rosters under ~25 creators, goat255/instagram-stories-highlights-scraper ($0.005-0.007/username, no start fee, no allowlist — verified) is cheaper; at 20 creators it is ~$18-25/mo vs $28.62, roughly a wash — pick on reliability (goat255 has only 137 users) and document via ADR either way.

### 4. Delete the redundant TikTok profile run — derive profile stats from the content run's authorMeta

- **Savings:** 50% of TikTok run count and $0.56/mo per creator Free tier ($0.48 Starter): 4 runs/day x ($0.001 start + 1 result x $0.0037) x 30d. Roster of 10 TikTok creators: ~$5.64/mo, plus halved telemetry rows. Verified: for profile-URL scrapes every video item's authorMeta already carries fans/heart/video/following/verified etc. by default ('this data is already included by default' per the scrapeAdditionalAuthorMeta doc).
- **Effort:** small code change + doc note (the SRC contract id keeps its data, just sourced from the content payload)
- **Changes:** app/Platform/Ingestion/Support/PollPlan.php + PollMonitoredAccountJob.php: skip IngestProfileJob for TikTok accounts; app/Platform/Ingestion/Providers/TikTok/TikTokContentAdapter.php:44-51: extract authorMeta from the first returned item and feed the existing IngestedProfileSync path (currently fed by TikTokProfileAdapter.php:41-48).
- **Risks / mitigation:** Quiet-day gap once recommendation 1 lands: the claim that a date-filtered run with zero matching videos still pushes a profile-info item rests on two changelog entries (verified quotes, but an inference not a README guarantee). Verify empirically on one dormant handle; if it fails, keep one profile run per day (cost: $0.14/mo/creator) so the follower time series never gaps more than 24h.

### 5. Consolidate reels into the post scraper (drop apify~instagram-reel-scraper) — gated on a cheap empirical A/B

> **RESOLVED 2026-07-07: A/B RUN — RECOMMENDATION REJECTED, the reel scraper stays.** Live test (zara/nike/natgeo, 15 items per actor per account, ~$0.24): the post scraper missed reels on all three accounts. Decisive evidence beyond feed-depth artifacts: zara's posts feed contained ZERO reels while 15 recent reels existed (grid-hidden reels are real), and two reels published INSIDE the post feed's own window (zara DZ5GKtj…/2026-06-22, natgeo DaZOSfk…/2026-07-05) never appeared among posts. Metric fields on the post scraper's Video items were complete (views/plays/likes/comments), so the failure is coverage, not fields. Both scrapers stay, each already windowed by rec 1 — do not revisit without new actor documentation.

- **Savings:** Up to $9.48/mo per IG creator Free tier ($8.40 Starter) — 49% of IG content spend at current defaults; proportionally less after recommendation 1. Basis (verified): the post scraper returns reels (type 'Video', productType 'clips') with videoViewCount, videoPlayCount, likesCount, commentsCount, and the codebase already maps them (InstagramPostAdapter.php:60-69 'Video' => ContentType::Reel) and dedupes on (platform, external_id) — today grid-shared reels are literally paid for twice ($0.0027 + $0.0026 per reel per cycle).
- **Effort:** small code change + needs ADR (removing an actor from the frozen SRC set is exactly what ADR-0001/0002 governance exists for)
- **Changes:** First run the A/B: same 3-5 roster creators through both actors at the same resultsLimit, compare shortCode sets (~$0.15 total). If coverage matches: remove the reel run from app/Platform/Ingestion/Jobs/IngestContentJob.php / PollPlan, retire InstagramReelAdapter.php, drop APIFY_ACTOR_INSTAGRAM_REEL (config/services.php:52-68); document in the decision log.
- **Risks / mitigation:** Two verified coverage gaps argue for the A/B first: trial reels are only documented on the reel scraper, and neither page documents whether grid-hidden reels appear in the posts feed. Also the reel scraper's sharesCount add-on has no post-scraper equivalent (it is paid, off by default, and unused today — confirm nothing in EmvCalculator expects shares). If the A/B shows misses, keep the reel scraper but apply recommendation 1's onlyPostsNewerThan to it instead.

### 6. Config-only stopgap while rec 1 is built: cut QDS_INGESTION_CONTENT_LIMIT from 30 to 15

- **Savings:** ~48% of content items immediately: per IG creator -1,800 posts ($4.86) - 1,800 reels ($4.68) = -$9.54/mo; per TikTok creator -1,800 results = -$6.66/mo (Free tier). Roster: ~$257/mo. Fully reversible.
- **Effort:** config-only (env change, no deploy)
- **Changes:** QDS_INGESTION_CONTENT_LIMIT=15 (config/qds.php:60, consumed at InstagramPostAdapter.php:51, InstagramReelAdapter.php:46, TikTokContentAdapter.php:49; YouTubeContentAdapter unaffected in cost terms).
- **Risks / mitigation:** Shrinks the metric-refresh window to the newest 15 items: a creator whose 16th-newest post is still accruing engagement stops getting refreshes for it (frozen EMV curve). Miss risk for new content is low (requires >15 posts between 6h cycles) but the refresh-window risk is real for prolific creators — treat as temporary and supersede with rec 1's date window, which bounds by TIME not COUNT and is the correct semantics for KPI freshness.

### 7. Adaptive cadence: demote dormant creators, gate story polls on recent story activity

- **Savings:** Roster-dependent, ~20-40% on top of the above for typical rosters where a third of creators are dormant in any given week. Example: a dormant IG creator at 1 full cycle/day instead of 4 saves 75% of their residual spend; story polls only for accounts with a story seen in the last 7 days (plus one daily probe) cuts story spend roughly in proportion to inactive share.
- **Effort:** code change + needs ADR (cadence is explicitly flagged 'NOT canonically decided' in config/qds.php comments — this is the moment to decide and document it)
- **Changes:** app/Platform/Ingestion/Jobs/RunMonitoringCycleJob.php / PollMonitoredAccountJob.php: add per-account due-logic (mirror the isDue pattern in app/Platform/Snapshots/DatabaseSnapshotScheduler.php:126-136) keyed on last-content-seen / last-story-seen timestamps already available from persisters; config keys for demoted cadence and story-activity window under qds.ingestion.*; exempt creators attached to an active campaign (Campaign/SeedingCampaign status) from demotion.
- **Risks / mitigation:** Cold-start gap: a dormant creator who resumes posting is caught up to 24h late (bounded by the daily probe); campaign-linked creators must never be demoted or campaign EMV/CPE windows will have degraded resolution at exactly the moments that matter. Also fix the on-demand path while here: RunCreatorCycleJob.php:71-75 ignores the roster platform filter — make it respect MonitoredSubject.platforms so a 'Run monitoring now' click cannot poll excluded platforms.

### 8. Move to Apify Starter and batch remaining runs per actor per cycle (async endpoint)

- **Savings:** Unit prices drop ~12-15% via the verified Bronze store discount: roster baseline $530 -> $453/mo, and the $29 plan fee is offset by its included $29 usage credit. Batching itself is verified cost-NEUTRAL on the per-result actors (no per-run fee on post/profile scrapers; only $0.001 starts on reel/TikTok, saving a token $0.12-0.24/mo/creator) — its real value is 20-50x fewer runs (fewer sync-timeout retries, less ProviderCall write amplification, fewer chances for the 4x retry multiplier to fire) and it is a prerequisite for rec 3. Starter is also the access floor for the story actor without allowlist begging.
- **Effort:** billing change + small-to-medium code change (reuses rec 3's async ApifyClient path) + doc note
- **Changes:** Subscribe to Starter ($29/mo). Adapters send roster arrays per cycle: 'usernames' (profile scraper), 'username' (post scraper), 'username' (reel scraper, if kept), 'profiles' (TikTok) — all verified arrays with per-profile resultsLimit semantics and no documented maxItems. Restructure Ingest*Job from per-account to per-cycle fan-in, persisting per-account results from the shared dataset. Do NOT migrate to the general apify~instagram-scraper as part of this: verified that one run cannot return multiple content types ('Each run produces one content type'), it is price-identical at Starter ($2.30/1k) and MORE expensive than the specialized actors at higher tiers ($1.50 vs $1.00/1k floor) — migration would churn the frozen SRC contracts for zero savings.
- **Risks / mitigation:** A failed batched run affects every creator in that cycle (mitigate: chunk rosters at ~25-50 handles/run); per-account error attribution and quarantine in ProviderCallRecorder needs rework; the 300s sync wall (verified) makes the async endpoint mandatory for batches — do not just raise APIFY_TIMEOUT_SECONDS and hope.

### 9. Retry and timeout hardening (tail-risk cap, not steady-state savings)

- **Savings:** Caps the worst-case spend multiplier: today a platform-wide Apify brownout multiplies a whole cycle's spend by up to 4x (tries=4 replaying whole jobs), and IG content retries re-run BOTH actors even when only one failed (up to 8x). Expected-value savings are a few percent; the point is bounding the blowup, which at roster scale is up to ~$50+ in a single bad day.
- **Effort:** small code change
- **Changes:** app/Platform/Ingestion/Jobs/IngestContentJob.php:101-107: retry only the provider(s) that transiently failed, not the full pair; app/Platform/Ingestion/Jobs/Concerns/IngestionJobBehaviour.php:55-72: consult ProviderHealthState before re-invoking a provider that is FAILING with a permanent error category; config/services.php:49: raise APIFY_TIMEOUT_SECONDS from 240 to >=330 for any remaining sync calls (a client-aborted run that completes server-side is still billed, then retried as a second billed run); expose tries/backoff as env instead of hardcoded.
- **Risks / mitigation:** None to KPI. Slightly slower recovery from genuinely transient single-provider failures if the health gate is too aggressive — scope the gate to permanent error categories (auth/access/not-found), leaving Timeout/RateLimited on the existing bounded backoff path.

## Do NOT do (KPI-corrupting cost cuts)

- Do NOT switch to pure incremental fetching (only-ever-new posts, no refresh window). Re-polling recent posts is the ONLY mechanism that refreshes public_metrics — the hourly DatabaseSnapshotScheduler reads stored DB values and never calls providers (confirmed in the audit's dedup analysis). Fetching each post exactly once would freeze every engagement curve at its first-seen value, silently corrupting EMV/CPE/CPM and every engagement-rate trend. The correct shape is a bounded refresh window (recommendation 1), never a pure new-items filter.
- Do NOT cut story cadence below 2-3 polls/day or fold stories back into the 6h full cycle only. Stories expire in 24h; at 1 poll/day a story posted just after a poll is caught once at up to 23h of staleness or, with any single run failure, missed entirely — an unrecoverable KPI gap (there is no re-fetch for expired stories). 6x/day is negotiable; ~3x/day is the safety floor, and the savings below that are pennies once batching (rec 3) makes each poll cost $0.159 for the whole roster.
- Do NOT flip the post scraper to dataDetailLevel=basicData to grab the 37% discount ($1.70 vs $2.70/1k Free) without an empirical field test. The basic-vs-detailed field split is verified as completely undocumented (the term appears only in the schema enum); if videoViewCount/videoPlayCount or comment metrics live behind 'post-details', reel and video KPIs silently degrade. Test on a handful of runs first; only then consider it.
- Do NOT drop the reel scraper before the shortCode A/B (recommendation 5's gate). Trial reels are verified as reel-scraper-only coverage, and grid-hidden reel coverage in the posts feed is undocumented on both store pages. Dropping it blind risks a class of reels vanishing from monitoring — undercounting exactly the high-signal experimental content brands care about.
- Do NOT replace content polling with the profile scraper's embedded 'latest 12 posts' even though one $0.0026 profile event nominally substitutes for ~$0.028 of post results. The embedded posts are verified to lack video view/play counts and reel-specific metrics — sufficient for existence, insufficient for engagement KPIs. Usable as a cheap activity PROBE for adaptive cadence (rec 7), never as the metrics source.
- Do NOT switch stories to a per-story-billed actor (e.g. automation-lab at $0.0023/story + sessionid cookie). Per-story billing re-charges every still-live story on every poll — at 6 polls/day a 24h story is billed ~6 times — and the mandatory Instagram session cookie puts a real account at ban risk, threatening the whole story pipeline. Per-username-billed actors (current datavoyantlab default, or goat255) are the only sane shape for repeated polling.
- Do NOT disable qds.snapshots.enabled or stretch the full-cycle cadence past ~12h to save money. Snapshots cost zero Apify dollars (they are DB-only), so disabling them saves nothing while gapping the append-only MetricSnapshot time series that EMV/engagement-rate trends and the /crm/results dashboard are built on; and cycles slower than 2/day degrade metric-growth resolution below what engagement-velocity KPIs need. If cadence changes at all, it needs the ADR the config comment already demands — and the savings should come from recs 1-7 instead, which cut 80-90% without touching time-series integrity.
- Do NOT migrate to the general apify~instagram-scraper expecting consolidation savings. Verified: one run returns exactly one content type (no posts+reels+profile merge), it is price-identical to the specialized actors at Starter ($2.30/1k) and more expensive at higher tiers ($1.50 vs $1.00/1k floor). It would churn the ADR-0001/0002 frozen SRC contracts and the adapter layer for zero or negative savings — the batching win it seems to offer is already available on the specialized actors' array inputs.
