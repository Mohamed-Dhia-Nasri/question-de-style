## For a fresh adversarial pass (checkboxes)

- [ ] **Roster restriction-flag vs guard parity.** `BrandRestrictionGuard::restrictedCreatorIdsForName` duplicates the `assertNotRestricted` matching in PHP (needle `mb_strtolower(trim(...))`, union across all BrandPreference rows, batched `whereIn`). Prove the bulk pre-filter and the per-attach `assertNotRestricted` can never disagree (esp. multibyte/eszett names — the ß→SS uppercase-round-trip bug was found and fixed in a test during C6; check the production matcher has no analogous fold). Restricted creators must still be blockable at save even if the flag is stale.
- [ ] **Bulk attach tenant stamping.** Every roster attach and copy-roster goes through `$owner->creators()->syncWithoutDetaching($ids)` so the pivot `tenant_id` is stamped by `withPivotValue`. Hostile-probe a foreign-tenant creator id injected into `selectedCreatorIds` (should be dropped by tenant scope + the count-mismatch guard) and confirm no un-scoped `DB::table` pivot insert exists anywhere in the new code. Same for the wizard's dual attach (campaign AND run).
- [ ] **Wizard single-transaction integrity + authorize coverage.** `CampaignWizard::commit()` creates client→brand→campaign→run→rosters in one `DB::transaction`. Verify: (a) every created type is re-authorized (`create` on Client/Brand/Campaign/SeedingCampaign) not just Campaign at mount; (b) a mid-transaction failure rolls back ALL of it (no orphan client); (c) `client_mode=new`+`brand_mode=existing` tamper is rejected; (d) an existing brand of another client is rejected; (e) restricted-creator skip names survive because there's no redirect.
- [ ] **CSV import row isolation + normalization + tenancy.** Confirm a handle conflict rolls back only its own row's creator (no orphan) and the loop continues; handles are trim+strip-one-`@`; within-file duplicate (platform,handle) first-wins; the existing-handle existence query is tenant-scoped (`PlatformAccount` model query, not raw). Note the **Monitoring enrollment volume**: every imported creator enrolls into Monitoring inside `createCreator` — a 200-row import silently enrolls 200 subjects on the active roster (cost implication; is 200 the right cap?). Also: `updatedUpload()` parses without an explicit authorize gate (read-only, no write, no cross-tenant read — judge acceptability).
- [ ] **Inline-create authorize matrix.** Every type × every host: `openInlineCreate`/`saveInlineCreate` authorize `create` on the right model; product/campaign require a brand context; the brand-inline "new client" sub-path is NOT `@can('create', Client)`-gated in the UI (server-side 403 backstop only — a brand-but-not-client creator dead-ends at submit). Confirm the server authorize is always present and the missing UI gate is only cosmetic.
- [ ] **Quick-action `?create=1` guard.** The four index `mount()` hooks call `$this->create()` only when `request()->boolean('create') && auth()->user()->can('create', Model)`. Confirm a crm.view-only user hitting `/crm/clients?create=1` renders the page without the modal and without a 403, and that `create()`'s own authorize still fires for anyone who reaches it.
- [ ] **UsersIndex role restriction.** Staff-only + edited-user's-current-role validation and dropdown. Confirm: an existing ClientViewer is still editable (name change) without a forced promotion; creating/demoting-to ClientViewer is rejected; the shared `roles` render array also feeds the table role-FILTER dropdown (accepted collateral — ClientViewer no longer a filter option, still reachable via `?roleFilter=` and visible unfiltered). Self-demotion guard for staff roles unaffected.
- [ ] **Overview query costs.** `CrmOverview::render()` runs ~5 counts + 3 attention queries + 2 `withCount` list queries on every load, all tenant-scoped. Check for N+1 in the active-work lists (status badge + counts) and whether the attention queries are index-friendly at scale.
- [ ] **Shipment progressive form defense-in-depth.** `save()` re-clears hidden tracking/delivery values after validation. Confirm a tampered payload (status Pending but tracking + delivered_at set) cannot persist those values, and the `delivered_at after_or_equal shipped_at` rule can't be tripped by a hidden-field remnant.
- [ ] **Copy lint recursion.** `CrmCopyLintTest` uses Symfony `Finder` (recursive, no depth cap) so the new `partials/` blade IS scanned — confirmed by the controller; re-verify none of the new wizard/checklist/CSV copy trips a banned pattern (esp. "Step N", "variant", curly apostrophes).
